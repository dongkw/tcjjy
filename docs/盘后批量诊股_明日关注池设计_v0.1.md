# 盘后批量诊股与明日关注池设计 v0.1

> 目标：把“盘后复盘和次日准备”做成每天可用的功能入口。  
> 当前只做功能设计，不写代码。

---

## 1. 定位

盘后批量诊股属于 `09 workflow` 的盘后准备流程。

它不是：

- 自动交易。
- 全市场高频扫描。
- AI 黑盒选股。
- 历史回放。

它是：

```text
盘后刷新数据
  -> 批量诊断持仓和自选股
  -> 生成明日重点关注股票
  -> 第二天盘前和盘中使用
```

第一版目标是把每天要看的股票组织清楚，不追求策略一步到位。

---

## 2. 与现有功能关系

已有能力：

- 自选股 tab 管理。
- 自选股批量刷新行情。
- 自选股筛选记录。
- 持仓行情刷新。
- 盘前持仓检查。
- 检查项生成决策。
- 已决策 tab。

本功能复用这些底层能力，但要新增一个更明确的产品入口：

```text
盘后批量诊股
明日关注池
```

区别：

| 功能 | 目的 |
|---|---|
| 自选股筛选 | 从一批自选股里找候选 |
| 盘前持仓检查 | 识别当前持仓风险 |
| 盘后批量诊股 | 盘后统一整理持仓和自选股，准备明天重点关注 |
| 明日关注池 | 第二天盘前/盘中优先看的股票清单 |

---

## 3. 输入范围

第一版默认输入：

```text
当前持仓
自选股 tab
已决策 tab
用户手动选择的 tab
```

不建议第一版全市场扫描。

项目里的股票代码拉取脚本可以用于：

- 维护基础股票池。
- 校验股票代码。
- 补股票名称。
- 后续支持全市场粗筛。

但第一版盘后诊股默认不跑全市场。

---

## 4. 盘后批量诊股流程

第一版流程：

```text
选择诊股范围
  ↓
读取已落库的收盘行情快照
  ↓
读取行情、技术指标、估值、持仓上下文
  ↓
运行透明规则诊断
  ↓
生成诊股结果
  ↓
生成明日关注池
  ↓
页面展示和人工处理
```

盘后诊股不负责实时抓行情。

行情采集应该在盘后数据准备步骤完成。收盘后，同一交易日的收盘价、涨跌幅、技术指标、估值快照都应该保持稳定。盘后诊股只读取数据库里的快照数据，避免把“数据采集时间”和“策略判断时间”混在一起。

数据缺失或数据日期不一致时：

- 单票标记 `DATA_GAP`。
- 不能编造价格。
- 缺失原因进入诊股结果。

如果需要重新拉取行情，应在盘后诊股之前单独执行数据采集或刷新动作，而不是在诊股过程中隐式刷新。

### 4.1 盘后数据准备

盘后数据准备是盘后诊股的前置步骤。

第一版页面：

```text
/post-market-data
```

职责：

- 选择范围：当前持仓、自选股 tab。
- 拉取收盘后行情并写入 `market_quotes`。
- 同步持仓市值、浮盈亏、仓位。
- 同步账户总资产、证券市值、现金比例。
- 记录数据准备批次和单票明细。

第一版表：

```text
post_market_data_prep_runs
post_market_data_prep_items
```

盘后诊股页面只消费最近已落库行情，并展示最近一次盘后数据准备批次。

## 5. 诊股分类

第一版分类：

```text
POSITION_RISK      持仓风险
BUY_CANDIDATE      买入候选
WATCH_TOMORROW     明日观察
DATA_GAP           数据不足
IGNORE             暂不关注
NEUTRAL            普通
```

说明：

- 持仓股优先识别风险。
- 非持仓股优先识别是否值得明天观察。
- 数据不足不能给强结论。
- 分类只是“明天看什么”，不是买卖指令。

---

## 6. 第一版规则

第一版规则先保持简单，后续通过回放和模拟盘迭代。

### 6.1 持仓股

关注：

- 缺买入逻辑。
- 缺证伪点或止损价。
- 接近止损。
- 跌破止损。
- 浮亏较大。
- 仓位过高。
- 接近目标价。
- 达到目标价。

输出可复用 `position_pre_market_checks` 的部分规则。

### 6.2 非持仓自选股

关注：

- 价格站上 MA20。
- MA20 不低于 MA60。
- 20 日涨幅处于健康区间。
- 当日涨跌过大提示风险。
- PE/PB 极端提示风险。
- 数据缺失降级。

第一版可复用当前 `watchlist_screening.py` 的透明规则。

## 7. 明日关注池

明日关注池是盘后诊股的核心输出。

建议数据结构：

```text
next_day_watch_items
```

字段：

```text
item_id
run_id
symbol
name
source_type          POSITION / WATCHLIST / DECISION
category             POSITION_RISK / BUY_CANDIDATE / WATCH_TOMORROW / DATA_GAP
priority             HIGH / MEDIUM / LOW
reason
current_price
reference_price
trade_date
next_trade_date
review_status
decision_id
created_at
payload_json
```

用途：

- 第二天盘前查看。
- 盘中触发扫描候选。
- 人工标记处理状态。
- 一键生成决策。

---

## 8. 处理状态

明日关注项需要人工处理状态。

状态建议：

```text
UNREVIEWED       未处理
VIEWED           已查看
WATCH_INTRADAY   盘中盯盘
NEED_DECISION    需要决策
DECISION_CREATED 已生成决策
DONE             已处理
IGNORE           忽略
DEFER            暂缓
```

第一版只记录状态，不自动下单。

## 9. 页面设计

建议新增页面：

```text
/post-market-diagnosis
```

页面区域：

1. 操作区
   - 选择 tab。
   - 勾选是否包含持仓。
   - 按钮：运行盘后诊股。

2. 诊股运行记录
   - 运行时间。
   - 股票数量。
   - 成功数量。
   - 失败数量。
   - 明日关注数量。

3. 明日关注池
   - 优先级。
   - 分类。
   - 股票代码。
   - 名称。
   - 现价。
   - 理由。
   - 处理状态。
   - 操作按钮。

操作按钮：

- 查看股票详情。
- 标记盘中盯盘。
- 生成决策。
- 已处理。
- 忽略。
- 暂缓。

## 11. 与盘前和盘中衔接

盘后：

```text
post_market_diagnosis_results
  -> next_day_watch_items
```

次日盘前：

```text
next_day_watch_items
  -> 盘前重点关注
  -> 可生成 decision_result
```

盘中：

```text
WATCH_INTRADAY / DECISION_CREATED
  -> trigger_price_items
  -> intraday trigger scan
```

第一版先做到页面可看、可标记、可生成决策。

## 12. 功能边界

允许：

- 抓取行情。
- 写入诊股记录。
- 写入明日关注池。
- 人工处理状态。
- 生成决策记录。

禁止：

- 自动买卖。
- 自动修改持仓数量。
- 自动修改成本价。
- 数据缺失时给强结论。
- AI 直接覆盖规则结果。

---

## 13. 第一版验收标准

第一版完成后应满足：

- 能从 Web 手动运行盘后批量诊股。
- 能选择至少一个自选股 tab。
- 能包含当前持仓。
- 能刷新行情。
- 能同步持仓和账户估值。
- 能生成诊股运行记录。
- 能生成明日关注池。
- 明日关注项能显示分类、优先级、理由和状态。
- 明日关注项能标记处理状态。
- 明日关注项能生成决策记录。
- 数据库校验通过。
- 不产生任何交易动作。

## 15. 与历史回放的关系

盘后批量诊股先解决日常使用闭环。

历史回放后续用于验证：

- 哪些盘后分类真的有价值。
- 明日关注池命中率如何。
- 关注后不买是否错过机会。
- 关注后买入是否有正收益。

所以顺序建议：

```text
先做盘后诊股功能
再用历史回放验证诊股规则
```
