# 13 Web Dashboard 界面设计 v0.1

> 本文档用于指导第一版本地 Web 界面开发。  
> 当前目标不是做复杂网站，而是把已有 CLI、SQLite、JSON、Markdown 结果变成一个能稳定使用的本地工作台。

---

## 1. 定位

第一版界面是：

```text
本地 Web Dashboard
```

运行方式：

```powershell
python .\web_dashboard.py
```

访问地址：

```text
http://127.0.0.1:8000
```

核心定位：

- 降低命令行使用门槛。
- 快速查看账户、持仓、决策、报告、回放、策略迭代记录。
- 让后续模拟盘和历史回放结果更容易复盘。
- 作为后续插件、AI 接口、任务调度的可视化入口。

第一版界面不是：

- 不是公网网站。
- 不是自动实盘交易平台。
- 不是策略黑盒。
- 不是复杂权限系统。
- 不是取代 CLI 的全部能力。

---

## 2. 核心原则

### 2.1 本地优先

第一版只监听：

```text
127.0.0.1
```

不开放公网访问，不做登录注册，不存外部用户数据。

### 2.2 读多写少

第一版以查看为主。

允许的轻操作：

- 刷新 SQLite 导入。
- 执行数据库校验。
- 执行数据库同步对账。
- 生成数据库备份。
- 打开本地报告。
- 维护持仓计划字段，例如买入逻辑、证伪点、止损价、目标价、计划仓位和备注。

暂不允许：

- 在界面里自动实盘下单。
- 在界面里直接绕过确认执行买卖。
- 在界面里直接修改策略核心规则。
- 在界面里修改现金、成本、持仓数量、可卖数量或 T+1 锁定数量。

### 2.3 界面不承担策略逻辑

策略、风控、模拟盘、回放继续由现有模块负责。

界面只做：

- 查询。
- 展示。
- 触发已有安全命令。
- 汇总状态。
- 链接报告。

任何交易结论必须能追溯到数据库、JSON 底稿和 Markdown 报告。

持仓计划字段的人工维护必须写入审计记录；该操作只补充策略上下文，不代表真实交易或券商账本变化。

### 2.4 服务策略迭代

界面的真正价值不是“好看”，而是更快发现策略问题。

必须优先展示：

- 哪些建议后来赚钱或亏钱。
- 哪些决策被风控拒绝。
- 哪些数据缺失导致降级。
- 哪些策略版本表现更好。
- 哪些错误需要进入调参记录。

### 2.5 时间清晰

所有页面必须明确展示：

- 当前自然日。
- 当前交易日。
- 数据更新时间。
- 决策时间。
- 报告生成时间。
- 最近一次 SQLite 导入时间。
- 数据库同步对账状态。

时间不清楚时，界面必须提示，而不是静默展示。

## 3. 技术选型

第一版建议：

```text
FastAPI + Jinja2 + SQLite + 原生 CSS + 少量原生 JS
```

原因：

- 和 Python 现有代码一致。
- 后续可平滑扩展 API。
- 不引入前端构建复杂度。
- 可在 IDEA 里直接运行。
- 能支撑第一版本地工作台。

第一版不建议：

- React / Vue：当前没有必要引入构建链。
- Streamlit：短期快，但后续做正式 API、插件和任务面板不如 FastAPI 顺。
- Electron：本地桌面壳太重。

后续如果页面复杂度明显上升，再考虑：

```text
FastAPI API + React/Vue 前端
```

## 4. 目录设计

新增目录建议：

```text
web_dashboard.py

src/ai_trader/web/
  __init__.py
  app.py
  routes.py
  services.py
  view_models.py
  commands.py
  settings.py
  templates/
    base.html
    dashboard.html
    accounts.html
    positions.html
    decisions.html
    reports.html
    replays.html
    strategy_iterations.html
    data_health.html
    settings.html
  static/
    css/
      app.css
    js/
      app.js
```

职责：

- `web_dashboard.py`：根目录兼容入口。
- `app.py`：创建 FastAPI 应用。
- `routes.py`：页面路由和 API 路由。
- `services.py`：读取 SQLite 和 Markdown，不写业务策略。
- `view_models.py`：把数据库行转换成页面需要的数据结构。
- `commands.py`：封装允许从界面触发的安全命令。
- `settings.py`：读取本地配置，例如数据库路径和输出目录。
- `templates/`：HTML 模板。
- `static/`：CSS 和少量 JS。

## 5. 数据来源

第一版界面主要读取：

```text
data/ai_trader.sqlite3
```

SQLite 来自：

```powershell
python .\database.py import-json
```

原始事实源仍然是：

```text
data/**/*.json
data/**/*.jsonl
data/reports/*.md
```

界面读取优先级：

1. SQLite 用于列表、摘要、筛选和关联查询。
2. Markdown 用于报告详情展示。
3. JSON/JSONL 用于详情页原始数据查看。

第一版不直接读取外部行情接口。

## 6. 页面规划

### 6.1 总览页 `/`

目标：打开界面后，一屏看到系统是否正常、今天该看什么。

展示内容：

- 数据库状态。
- 最近一次导入时间。
- 数据校验状态。
- 账户总资产、现金、持仓市值、仓位比例。
- 当前持仓数量。
- 今日/最近决策数量。
- 最近风控拒绝数量。
- 最近回放数量。
- 最近策略调参记录。
- 最近报告列表。

重点提示：

- 数据库不存在。
- 数据库未导入。
- 校验存在 ERROR。
- 最近数据时间过旧。
- 今天非交易日。

第一版操作：

- 刷新导入。
- 执行校验。
- 生成备份。

### 6.2 账户页 `/accounts`

目标：看清本金、现金、仓位和账户变化。

展示内容：

- 账户列表。
- 初始资金。
- 可用现金。
- 冻结资金。
- 持仓市值。
- 总资产。
- 今日买入已用额度。
- 今日卖出金额。
- 现金比例。
- 权益仓位。
- 最近账户快照。

后续增强：

- 资产曲线。
- 回撤曲线。
- 与基准对比。

### 6.3 持仓页 `/positions`

目标：看清当前持仓和卖出约束。

展示内容：

- 股票代码。
- 名称。
- 资产类型。
- 总数量。
- 可卖数量。
- T+1 锁定数量。
- 成本价。
- 当前价。
- 市值。
- 浮盈亏。
- 仓位比例。
- 买入逻辑。
- 证伪点。
- 止损价。
- 最近交易日。
- 状态。

重点提示：

- `locked_quantity > 0` 时标记 T+1 锁定。
- 当前价缺失时标记数据不足。
- 持仓没有买入逻辑或证伪点时标记复盘缺口。

### 6.4 决策页 `/decisions`

目标：查看每次买入、持仓、卖出评估。

筛选条件：

- 股票代码。
- 任务类型。
- 最终动作。
- 置信度。
- 策略版本。
- 日期区间。
- 是否需要人工复核。

列表字段：

- 决策时间。
- 交易日。
- 股票代码。
- 名称。
- 任务类型。
- 最终动作。
- 置信度。
- 策略版本。
- 数据质量。
- 是否人工复核。
- 报告链接。

详情展示：

- 决策结论。
- 行动理由。
- 触发价。
- 关联 strategy_snapshot。
- 原始 JSON。
- 关联报告。

### 6.5 风控与组合页 `/plans`

目标：查看哪些股票被允许买、哪些被拒绝、钱怎么分配。

展示内容：

- 风控检查记录。
- 阻塞规则。
- 警告规则。
- 组合计划。
- 买入预算。
- 候选数量。
- 拒绝数量。
- 计划买入金额。
- order_intents 列表。

重点提示：

- 被风控拦截的原因必须明显。
- 计划金额不能超过账户约束。
- `READY_FOR_CONFIRM` 和 `REJECTED` 要分清。

### 6.6 工作流页 `/workflows`

目标：查看盘前、盘中、盘后流程执行结果。

展示内容：

- workflow_runs。
- 盘前计划。
- 触发价列表。
- 盘中扫描记录。
- 盘中触发事件。
- 盘后报告。

筛选条件：

- 交易日。
- 工作流类型。
- 状态。
- 账户。

重点提示：

- 工作流失败。
- 非交易日运行。
- 盘中触发数量。
- 重复触发数量。

### 6.7 历史回放页 `/replays`

目标：查看回放实验结果。

展示内容：

- replay_id。
- 股票范围。
- 开始日期。
- 结束日期。
- 初始资金。
- 最终资产。
- 总收益率。
- 最大回撤。
- 交易次数。
- 胜率。
- 盈亏比。
- 策略版本。
- 回放报告链接。

详情展示：

- 每日回放记录。
- 每日资产。
- 买入/卖出次数。
- 阻塞原因。
- 绩效指标。

后续增强：

- 收益曲线。
- 回撤曲线。
- 策略版本对比。

### 6.8 策略迭代页 `/strategy-iterations`

目标：把回放和模拟盘中的问题沉淀为调参记录。

展示内容：

- iteration_id。
- 来源类型。
- 来源 ID。
- 策略版本。
- 上一策略版本。
- 区间。
- 收益率。
- 最大回撤。
- 交易次数。
- 自动识别问题。
- 人工记录问题。
- 假设。
- 规则改动。
- 风控改动。
- 仓位改动。

后续可增加：

- 新增调参记录表单。
- 策略版本差异对比。
- 固定样本回归入口。

### 6.9 报告页 `/reports`

目标：集中查看所有 Markdown 报告。

报告类型：

- 单票决策报告。
- 模拟盘报告。
- 组合计划报告。
- 盘前计划报告。
- 盘中触发报告。
- 盘后复盘报告。
- 回放报告。

列表字段：

- 报告类型。
- 标题。
- 股票代码。
- 交易日。
- 生成时间。
- 文件路径。

详情展示：

- 第一版可以直接展示 Markdown 原文。
- 后续再做 Markdown 渲染。

### 6.10 数据健康页 `/data-health`

目标：让用户知道当前数据是否可信。

展示内容：

- 数据库是否存在。
- 最近导入批次。
- 导入数量。
- 导入失败信息。
- `reconcile` 同步对账状态。
- 当前文件数量和 SQLite 索引行数。
- 当前文件未索引到 SQLite 的问题列表。
- 最近 `db_sync_errors.jsonl` 同步错误。
- `validate` 问题列表。
- 文件索引数量。
- 决策和快照引用缺失。
- 成交和订单引用缺失。
- 持仓锁定数量不一致。

问题分级：

```text
ERROR    必须先处理
WARNING  可以继续看，但不能盲目交易
INFO     普通提示
```

### 6.11 设置页 `/settings`

目标：查看本地运行配置。

第一版展示：

- output_dir。
- db_path。
- Python 版本。
- 项目路径。
- 当前时区。
- 监听地址。
- 监听端口。

第一版不做敏感配置输入。

## 8. 命令触发边界

界面第一版只允许触发安全命令：

```powershell
python .\database.py import-json
python .\database.py validate
python .\database.py reconcile
python .\database.py backup
```

不在第一版界面触发：

```powershell
python .\paper_trading.py apply --confirm
python .\workflow.py intraday
python .\historical_replay.py run
```

原因：

- 这些命令会产生业务结果。
- 有些命令可能耗时。
- 有些命令涉及交易语义。
- 第一版先把查看和复盘做好。

后续如果要加操作按钮，必须单独设计确认流程、任务状态、失败重试和操作审计。

## 10. 状态标记规范

建议统一状态样式：

```text
OK          正常

ERROR       有错误
STALE       数据过旧
MISSING     数据缺失
BLOCKED     被规则阻塞
READY       可确认
RECORDED    仅记录
```

交易动作标签：

```text
BUY

HOLD
REDUCE_HALF
REDUCE_TO_WATCH
CLEAR
WAIT
DO_NOT_BUY
DATA_BLOCKED
NO_SELL_T_PLUS
```

页面必须避免只用颜色表达状态，状态文字也要显示。

## 11. 第一版验收标准

第一版完成后，应满足：

- 可以启动本地 Web 服务。
- 浏览器打开 `/` 后能看到数据库摘要。
- 没有数据库时给出明确提示。
- 可以从界面触发导入、校验、备份。
- 账户页能展示账户资金。
- 持仓页能展示 T+1 锁定和可卖数量。
- 决策页能筛选和查看决策。
- 报告页能打开 Markdown 报告。
- 数据健康页能展示 validate 问题。
- 数据健康页能展示 reconcile 同步对账状态。
- 顶部状态栏能展示同步 OK / ERROR / MISSING。
- 历史回放页能查看回放绩效。
- 策略迭代页能查看调参记录。
- 界面不包含任何自动实盘交易入口。
- 核心策略逻辑仍在后端模块，不在 HTML/JS 中实现。

## 14. 风险与约束

### 14.1 数据不是实时行情

第一版界面展示的是 SQLite 和本地文件数据，不等于实时行情终端。

如果数据未刷新，界面必须提示。

### 14.2 Markdown 报告可能过期

报告页必须显示生成时间。

过旧报告不能当作当前交易依据。

### 14.3 操作按钮要谨慎

任何会改变账本或生成交易结果的操作，都必须：

- 明确显示影响范围。
- 二次确认。
- 写入操作记录。
- 可以追溯。

第一版先不做这类按钮。

### 14.4 界面不能掩盖策略问题

界面只负责让问题更明显。

如果策略规则不足、数据质量差、回放样本少，界面必须如实暴露，不能通过视觉效果让结论显得更可靠。

## 15. 第一版结论

第一版 Web Dashboard 应该先做成：

```text
本地、轻量、可查询、可复盘、可校验
```

它的核心价值是：

```text
把已有交易辅助流程从命令行结果，变成一个可以持续观察和复盘的工作台。
```

第一版先把数据看清楚，再考虑把更多操作搬进界面。
