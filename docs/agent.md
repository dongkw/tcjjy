# 股票决策 Agent 执行说明

> 本文档定义本目录下股票买入/卖出评估的默认处理逻辑。  
> 核心原则：先用脚本拿结构化数据，再按决策引擎判断；不要直接凭印象下结论。

## 2. 用户意图识别

### 买入评估

用户表达类似：

- `<股票代码> 现在可以买吗`
- `<股票代码> 买入`
- `帮我看看能不能买`

执行模式：决策引擎模式一【买入评估】，走第 1-8 节。

### 持仓卖出/持有评估

用户表达类似：

- `<股票代码> 卖出`
- `<股票代码> 要不要卖`
- `<股票代码> 持有还是清仓`

执行模式：决策引擎模式二【持仓评估】，走第 9 节。

卖出模式必须提醒用户：完整判断需要成本价、当前盈亏%、当前仓位%、当初买入逻辑、原设证伪点。缺这些时，只能给“基于当前数据的预评估”。

## 3. 数据抓取流程

收到股票代码后，优先执行：

```powershell
python .\stock_decision_data.py <股票代码> --period short
```

如果用户明确说中线或长线：

```powershell
python .\stock_decision_data.py <股票代码> --period middle
python .\stock_decision_data.py <股票代码> --period long
```

脚本会生成：

```text
data/stock_json/stock_data_<股票代码>.json
```

后续评估优先读取该 JSON，不重复联网查同一批基础数据。

如果脚本部分接口失败：

1. 先分开抓行情和 K 线。
2. 再重试财务接口。
3. 仍失败时，明确标注数据缺口。
4. 不得编造缺失数据。

## 4. JSON 数据使用口径

### 行情与估值

使用字段：

- `quote.price`
- `quote.pe_ttm`
- `quote.pb`
- `quote.dividend_yield_pct`
- `quote.market_cap_yuan`
- `quote.pct_change`

用途：

- 当前股价。
- PE/PB/股息率。
- 大跌日或日内风险判断。
- 仓位与止损计算基准。

### 技术面

使用字段：

- `technical.ma20`
- `technical.ma60`
- `technical.ma20_slope_up`
- `technical.ma60_slope_up`
- `technical.above_ma20`
- `technical.above_ma60`
- `technical.change_20d_pct`
- `technical.change_60d_pct`
- `technical.high_20d`
- `technical.low_20d`
- `technical.atr14_pct`
- `technical.recent_10d`

短线核心：

- 股价站上 20 日线。
- 20 日线向上。
- 未触发大跌日禁追。

中线核心：

- 股价站上 60 日线。
- 60 日线向上。

### 财务数据

使用字段：

- `financial.latest`
- `financial.annual_last_3`
- `financial.checks`

重点检查：

- 最近两个会计年度扣非净利润是否均为负。
- 经营现金流/净利润是否连续 3 年低于 0.5。
- 最新季度收入、归母净利润、扣非净利润同比。
- 最新季度经营现金流是否明显恶化。
- 近 3 年 ROE、毛利率、净利率、资产负债率。

### 机械标记

使用字段：

- `engine_flags.a2_deducted_profit_negative_2y`
- `engine_flags.c3_ocf_low_3y`
- `engine_flags.short_term_above_ma20_and_rising`
- `engine_flags.middle_term_above_ma60_and_rising`
- `engine_flags.chase_high_1m_over_80_pct`
- `engine_flags.chase_high_1m_50_to_80_pct`
- `engine_flags.big_down_day_over_5_pct`

这些字段只能作为机械判断，不等于最终结论。

## 5. 买入评估逻辑

严格按决策引擎 A-G 顺序执行。

### A 前置硬否决

直接检查：

- `engine_flags.a2_deducted_profit_negative_2y`
- `engine_flags.chase_high_1m_over_80_pct`
- 公告关键词：减持、监管、立案。

命中硬否决时，直接输出“不买入”。

### B 周期匹配

如果用户未说明周期：

- 默认按中线预评估。
- 最终建议最高给”观察仓/小仓”，不得直接给满仓”买入”。
- 若其余闸门全部通过，输出”补充周期后可复核买入”。

如果用户说短期/短线：

- 必须检查未来 1 个月催化剂。
- 技术面优先看 20 日线。

如果用户说中线：

- 必须能写出 1-2 年利润/现金流改善的量化或区间假设 + 跟踪指标。
- 技术面优先看 60 日线。

### C 基本面硬否决

重点检查：

- 扣非净利润是否连续为负。
- 归母净利润规模是否过小。
- 经营现金流/净利润是否连续 3 年 < 0.5。
- 最新季度现金流是否异常。
- 是否有监管、造假、非标意见。

### D 类型与估值

先判断类型：

- 稳定价值股。
- 成长股。
- 周期股。
- 困境反转股。

非周期股（稳定价值股/成长股）估值需要：

- 当前 EPS。
- 未来第 3 年 EPS 假设。
- 当前 PE。
- 目标 PE（近 5 年 PE 中位数，或行业合理区间）。
- 股息率。

周期股估值（不使用 PE 公式，按决策引擎 5.3 节）：

- 当前 PB。
- 近 10 年 PB 分位（理想情况 ≤ 30%）。
- 当前利润/ROE 是否处于历史高位。
- 产品价格处于周期低位/中位/高位。
- 行业库存、产能、亏损面。

周期股 PB 分位数据处理规则：

- 如果近 10 年 PB 分位可通过联网查询获取 → 使用实际分位值，≤ 30% 为达标。
- 如果 PB 分位无法获取但 PB < 1.2 且 ROE > 8% → 可定性判定为”估值处于低位区域”，D 闸门通过，但最终仓位上限下调 1/3，并在数据缺口中标注。
- 如果 PB > 2.0 或当前利润处于历史高位 → 即使分位缺失，也按”估值不达标”处理，最高”观望”。

估值结果与买入门槛（决策引擎 5.5 节）：

| 标的类型 | 预期年化回报门槛 | 判定 |
|----------|-----------------|------|
| 普通股票 | ≥ 8% | 达标→通过；5-8%→观望；<5%→不买入 |
| 高波动/小盘/强周期/困境反转 | ≥ 12% | 达标→通过；8-12%→观望；<8%→不买入 |

非周期股如果缺近 5 年 PE 中位数或未来第 3 年 EPS，不能直接给”买入”，但可反算”达到门槛所需买入价”或”达到门槛所需未来 EPS 增速”，作为观望触发条件。

### E 筹码风险

当前脚本只半覆盖公告关键词。  
股东户数、融资余额、北向资金、解禁缺失时，必须写入数据缺口。

### F 三段式逻辑

必须写清：

- 核心逻辑。
- 验证催化剂。
- 证伪信号。

证伪信号至少包含：

- 基本面证伪。
- 价格/技术证伪。
- 事件证伪。

写不出证伪信号时，不得给“买入”。

### G 技术与仓位

技术面负责买点确认和风险定位，不负责证明公司好坏。

**买点分四类（与决策引擎 8.1 节一致）：**

| 买点类型 | 条件 | 仓位动作 |
|----------|------|----------|
| 回踩买点 | 回踩 20/60 日线或平台下沿后缩量止跌 | 建首仓 |
| 突破买点 | 放量突破平台/前高，且次日不跌回平台 | 建首仓或加仓 |
| 修复买点 | 大跌后重新站回关键均线并连续 2 日企稳 | 小仓转正常首仓 |
| 左侧试错 | 基本面达标（A-F 通过）但技术未确认 | 不超过计划仓位 1/3 |

**按周期选参考均线：**

短线优先看 20 日线，中线优先看 60 日线。

- 趋势已成立（站上均线 + 均线向上）：可用回踩/突破/修复买点，建正常首仓。
- 趋势未成立但 A-F 已通过：可用**左侧试错**买点，仓位不超过计划仓位的 1/3，止损严格设在近 20 日低点或 1.5 倍 ATR。
- 均线空头排列且加速下行、无止跌迹象：即使是左侧试错也应等待更明确的止跌信号（如缩量止跌、下影线、底分型）。

**大跌日禁追规则：**

- 当日跌幅 >5% 且收盘接近日内低位，或成交额明显放大 → 不得给”立即买入”，最高”观察仓”。
- 跌破 20 日线且未收回 → 短线趋势买点不成立，但左侧试错仍可用（仓位降级）。
- 跌破 60 日线且未收回 → 中线趋势买点不成立，但左侧试错仍可用（仓位降级）。

**仓位执行细则：**

- 标准首仓 = 最终单票仓位的 1/2。
- 左侧试错或大跌日试错 = 最终单票仓位的 1/3。
- 加仓只能在”逻辑验证 + 技术确认”之后；价格下跌不是加仓理由。
- 单票仓位原则上 ≤ 30%，高波动/周期股自动下调。

## 6. 卖出/持有评估逻辑

卖出模式优先走决策引擎第 9 节。

### 必需持仓信息

完整判断需要用户提供：

- 成本价。
- 当前盈亏%。
- 当前仓位%。
- 当初买入逻辑。
- 原设证伪点。

缺失时，结论必须标注为“预评估”。

### 判断优先级

1. 原买入逻辑是否已经不成立。
2. 原证伪点是否触发。
3. 如果现在空仓，是否仍愿意以当前价重新买入。
4. 技术止损位是否跌破。
5. 是否出现暴涨止盈条件。

### 短线卖出规则

短线/波段以价格止损优先：

- 跌破 20 日线且收不回：减仓一半。
- 跌破近 20 日低点：继续减仓或清仓。
- 反弹未站回 20 日线：不加仓，最多持有观察仓。
- 站回 20 日线并连续 2 日不跌回：剩余仓可观察。

### 中线卖出规则

中线看 60 日线：

- 跌破 60 日线且收不回：趋势走弱，减仓。
- 站回 60 日线且均线向上：趋势修复。

### 重新买入测试

卖出评估必须回答：

> 如果现在空仓，是否仍愿意以当前价重新买入？

如果答案是否：

- 至少减仓。
- 若同时跌破关键技术位或逻辑证伪，清仓。

## 7. 输出格式

### 买入评估输出

各档位定义：

- **买入**：A/C/E/F 无硬否决，B 周期匹配，D 估值达标，G 趋势买点成立；给满仓建议。
- **观察仓（小仓试错）**：A/C 通过，D/F 达标或接近达标，但 G 趋势未确认（左侧试错场景），或数据有缺口但逻辑支撑较强，或周期未确认但其余通过；仓位不超过计划仓位 1/3，必须给出明确的加仓触发条件和止损。
- **观望**：基本面未硬否决，但估值不达标、关键数据大量缺失、或技术面无任何止跌信号；必须给出触发买入的价格/事件。
- **不买入**：任一硬否决闸门命中。

必须包含：

- 【最终建议】买入 / 观察仓（小仓试错） / 观望 / 不买入
- 【一句话结论】
- 【信心评级】
- 【关键数据】
- 【决策依据（按闸门顺序）】
- 【若买入/观察仓，执行参数】
- 【风险提示与数据缺口】

### 卖出评估输出

必须包含：

- 【最终建议】持有 / 减仓一半 / 清仓
- 是否为缺持仓信息下的预评估。
- 当前价、20日线、60日线、近20日高低点。
- 基本面是否触发明显证伪。
- 具体执行价位。

卖出执行价位常用口径：

- 20日线：短线减仓/修复线。
- 60日线：中线趋势线。
- 近20日低点：前低失守线。
- 近20日高点：反弹压力或突破确认线。

## 8. 数据缺口处理

任何缺失数据都必须明说。

常见缺口：

- 成本价。
- 当前仓位。
- 原买入逻辑。
- 原证伪点。
- 近5年 PE 中位数。
- 未来第3年 EPS 假设。
- 股东户数。
- 融资余额。
- 北向资金。
- 解禁。
- 公告原文复核。
- 应收账款/营收。
- 存货/营收。
- 商誉/净资产。

关键数据缺失时，分级降档（不搞一刀切封顶）：

- 缺少 1-2 个关键数据（如仅缺 PE 中位数或 PB 分位）→ 结论降一档（买入→观察仓，观察仓→观望），并在缺口中写明反算条件。
- 缺少 3 个及以上关键数据 → 买入评估最高”观望”，卖出评估最高”预评估”。
- 周期股缺 PB 分位但 PB 绝对值可获取 → 按 G 闸门周期股定性规则处理，不视为硬数据缺口。
- 基本面数据（营收、利润、OCF/NP）缺失 → 属于硬缺口，必须补齐后才能评估，当前直接给”数据不足，暂不评估”。
- 筹码数据（股东户数/融资/北向/解禁）→ 属于次要缺口，不封顶但必须在风险提示中列出，结论中注明”筹码数据未经完整核验”。
- 卖出评估只要缺成本价/仓位/原买入逻辑 → 标注为”基于当前数据的预评估”。

---

## 9. 默认纪律

- 不预测股价，只判断是否符合规则。
- 不因为估值低就忽略趋势破位。
- 不因为趋势破位就忽略估值低 — 左侧试错是允许的，但仓位必须降级且严格止损。
- 不因为基本面好就无视证伪点。
- 不因为价格下跌就自动补仓。
- 不因”还会涨”加仓。
- 脚本结果和机械 flag 只做数据输入，不直接替代判断。
- 最终结论必须能对应到具体数字和规则。
