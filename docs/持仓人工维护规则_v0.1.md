# 持仓人工维护规则 v0.1

> 本文档用于收口 Web 持仓详情页的人工维护边界，避免界面写入偏离架构方向。

---

## 1. 定位

持仓详情页允许维护的是“持仓计划信息”，不是交易执行。

允许维护：

- 原买入逻辑。
- 证伪点。
- 止损价。
- 目标价。
- 计划仓位。
- 备注。

不允许通过该页面：

- 修改真实成交。
- 修改可卖数量。
- 修改锁定数量。
- 修改成本价。
- 修改现金。
- 生成买入或卖出委托。

---

## 2. 数据边界

当前系统仍处于 v0.1 阶段：

```text
JSON/JSONL 仍是多数业务账本的事实源
SQLite 是查询索引 + 部分本地维护覆盖层
```

持仓人工计划字段暂时以 SQLite 为准：

```text
buy_logic
invalidation_point
stop_loss_price
target_price
planned_position_pct
position_note
```

重新执行 `database.py import-json` 时，不应覆盖 SQLite 中已经人工维护过、而 JSON 中为空的计划字段。

---

## 3. 审计要求

每次人工维护持仓计划，必须写入 `audit_logs`。

审计至少记录：

- 修改时间。
- 账户。
- 股票。
- 操作类型。
- 修改前字段。
- 修改后字段。
- 操作来源。

如果表单提交没有实际字段变化，可以不新增审计记录。

---

## 4. 后续演进

后续如果 SQLite 成为主账本，应把持仓计划信息纳入正式写入模型。

如果继续保留 JSON 归档，应增加导出机制：

```text
SQLite 持仓人工计划字段 -> JSON 归档
```

在此之前，人工维护字段不能被视为券商事实，只能作为策略判断和复盘使用的本地上下文。
