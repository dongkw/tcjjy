# 股票决策数据底稿字段说明与规则映射

> 本文档解释 `stock_decision_data.py` 生成的 JSON 如何服务两份规则文件：  
> - 《股票投资决策引擎（AI执行版 v2.0）》  
> - 《股票投资分析自查手册（逻辑修订版 v2.0）》  
>
> 第一份文档《股票决策数据底稿（JSON通用模板）》偏“填什么”；本文档偏“为什么填、怎么用”。

---

## 1. 整体流程

推荐流程：

1. 用户给股票代码和周期。
2. 执行脚本生成 JSON：`python .\stock_decision_data.py <股票代码> --period short|middle|long`
3. 用《股票决策数据底稿（JSON通用模板）》整理 JSON。
4. 用本文档检查字段和规则映射。
5. 再进入《股票投资决策引擎》按 A-G 闸门输出最终结论。

核心原则：

- JSON 是数据层。
- 通用模板是整理层。
- 字段说明是解释层。
- 决策引擎是判断层。
- 自查手册是人工复核层。

---

## 2. 与决策引擎输入清单的对应关系

| 决策引擎输入项 | JSON可提供字段 | 是否完整 | 说明 |
|----------------|----------------|----------|------|
| 股票名称/代码 | `quote.name`, `quote.code` | 是 | 可直接使用 |
| 当前股价 | `quote.price` | 是 | 可直接使用 |
| 用户持有周期意图 | `meta.requested_period` | 部分 | 来自脚本参数，但仍需确认用户真实意图 |
| 近3年营收 | `financial.annual_last_3[].revenue_yuan` | 是 | 单位为元，展示时建议换算为亿元 |
| 近3年归母净利润 | `financial.annual_last_3[].parent_net_profit_yuan` | 是 | 单位为元 |
| 近3年扣非净利润 | `financial.annual_last_3[].deducted_net_profit_yuan` | 是 | 单位为元 |
| 毛利率 | `financial.*.gross_margin_pct` | 是 | 最新一期和近3年年报均可用 |
| 净利率 | `financial.*.net_margin_pct` | 是 | 最新一期和近3年年报均可用 |
| ROE | `financial.*.roe_pct` | 是 | 最新一期和近3年年报均可用 |
| 资产负债率 | `financial.*.debt_asset_ratio_pct` | 是 | 最新一期和近3年年报均可用 |
| 近3年经营现金流/净利润 | `financial.annual_last_3[].ocf_to_net_profit` | 是 | C-3核心字段 |
| 自由现金流 | `financial.*.free_cashflow_yuan` | 部分 | 口径取决于数据源 |
| 应收账款/营收 | 暂无 | 否 | 需后续扩展或人工补充 |
| 存货/营收 | 暂无 | 否 | 需后续扩展或人工补充 |
| 商誉/净资产 | 暂无 | 否 | 需后续扩展或人工补充 |
| 当前PE-TTM | `quote.pe_ttm` | 是 | 非周期股估值字段 |
| 近5年PE中位数 | 暂无 | 否 | 需要另行补充 |
| 当前PB | `quote.pb` | 是 | 周期股估值字段 |
| 近10年PB分位 | 暂无 | 否 | 需要另行补充 |
| 当前股息率 | `quote.dividend_yield_pct` | 是 | 回报率公式字段 |
| 未来第3年EPS预估 | 暂无 | 否 | 必须人工假设 |
| 减持/监管/回购 | `announcements.*` | 部分 | 关键词初筛，需人工复核公告原文 |
| 股东户数/融资/北向/解禁 | 暂无 | 否 | 需后续扩展或人工补充 |

## 3. 闸门A：前置硬否决映射

| 规则 | JSON字段 | 自动化程度 | 使用方法 |
|------|----------|------------|----------|
| A-1 大股东/实控人近3月大幅减持 | `announcements.减持` | 半自动 | 只能按标题初筛，必须人工看原公告判断比例和原因 |
| A-2 最近两个会计年度扣非净利润均为负 | `engine_flags.a2_deducted_profit_negative_2y` | 自动 | true则普通股直接触发硬否决 |
| A-3 监管立案或非标审计 | `announcements.监管` | 半自动 | 关键词初筛后人工核对原文和审计意见 |
| A-4 近1个月暴涨 >80% 且无基本面改善 | `engine_flags.chase_high_1m_over_80_pct` | 部分自动 | 涨幅可自动算，是否有基本面改善需人工判断 |

注意：

- A-1 和 A-3 不能只看关键词结果。
- A-4 需要结合最新财报和事件判断，脚本只能算涨幅。

---

## 4. 闸门B：周期与意图匹配映射

| 周期 | JSON字段 | 能自动支持什么 | 仍需人工判断什么 |
|------|----------|----------------|------------------|
| 短线/波段 | `meta.requested_period`, `technical.*`, `financial.recent_reports` | 判断是否处于20日线可操作区、近期是否有财报窗口 | 未来1个月催化剂是否明确 |
| 中线 | `financial.latest`, `financial.annual_last_3`, `technical.ma60` | 判断利润和现金流是否改善、60日趋势是否确认 | 未来1-2年量化假设和跟踪指标 |
| 长线 | `financial.annual_last_3` | 初步看盈利质量、ROE、现金流 | 商业模式、竞争格局、长期复利基础 |

周期字段只是用户输入的记录，不等于逻辑一定匹配。  
最终仍要回答：这个周期下，靠什么赚钱，何时验证，错了怎么办。

## 5. 闸门C：基本面硬否决映射

| 规则 | JSON字段 | 自动化程度 | 说明 |
|------|----------|------------|------|
| C-1 扣非净利润连续2年低于净利润50% | `financial.annual_last_3` | 可扩展 | 当前脚本未单独输出该flag，但数据已具备 |
| C-2 年归母净利润 < 5000万 | `financial.annual_last_3[].parent_net_profit_yuan` | 可人工快速判断 | 小公司需结合主营稳定增长证据 |
| C-3 经营现金流/净利润连续3年 < 0.5 | `engine_flags.c3_ocf_low_3y` | 自动 | true则触发硬否决 |
| C-4 应收/存货异常上升 | 暂无 | 未覆盖 | 需扩展资产负债表字段 |
| C-5 商誉/净资产过高 | 暂无 | 未覆盖 | 需扩展商誉和净资产字段 |
| C-6 监管处罚、造假、非标意见 | `announcements.监管` | 半自动 | 需人工复核 |

当前脚本最强的是 C-3 现金流硬规则；C-4、C-5 后续应作为扩展重点。

## 6. 闸门D：类型与估值映射

### 非周期股

回报率公式需要：

| 参数 | JSON字段 | 当前状态 |
|------|----------|----------|
| 当前EPS | `financial.latest.eps` 或最近年报EPS | 有 |
| 未来第3年EPS | 暂无 | 需人工假设 |
| 当前PE | `quote.pe_ttm` | 有 |
| 目标PE | 近5年PE中位数 | 缺 |
| 当前股息率 | `quote.dividend_yield_pct` | 有 |

因此 JSON 可以启动估值，但不能独立完成估值通过结论。

### 周期股

周期股需要：

| 参数 | JSON字段 | 当前状态 |
|------|----------|----------|
| 当前PB | `quote.pb` | 有 |
| 近10年PB分位 | 暂无 | 缺 |
| 利润/ROE是否历史高位 | `financial.annual_last_3`, `financial.latest` | 部分 |
| 产品价格/库存/产能 | 暂无 | 缺 |

周期股判断仍需要行业数据，不能只靠公司 JSON。

## 7. 闸门E：筹码风险映射

| 信号 | JSON字段 | 当前覆盖 |
|------|----------|----------|
| 大股东无合理解释大幅减持 | `announcements.减持` | 半覆盖 |
| 股东户数本季暴增 >30% | 暂无 | 未覆盖 |
| 融资余额连升而股价滞涨 | 暂无 | 未覆盖 |
| 北向资金持续净流出 | 暂无 | 未覆盖 |
| 未来1月大量解禁 | 暂无 | 未覆盖 |
| 高管/公司真实回购 | `announcements.回购` | 半覆盖 |

筹码数据目前是脚本短板。  
最终判断时，若这些数据缺失，应在风险提示里明确写出，不能假装已经检查完整。

---

## 8. 闸门F：三段式逻辑映射

闸门F不能由 JSON 自动完成。

| 必填项 | JSON能提供的辅助 | 必须人工完成的判断 |
|--------|------------------|--------------------|
| 核心逻辑 | 收入、利润、毛利率、现金流趋势 | 市场误解在哪里，优势信息是什么 |
| 验证催化剂 | 最新财报、同比增速、公告线索 | 最晚何时、用什么事件或数据验证 |
| 证伪信号 | 技术位、现金流、利润增速 | 基本面证伪、价格证伪、事件证伪 |

如果证伪信号写不出来，决策引擎规定最终只能否决。

## 9. 闸门G：技术面与仓位映射

| 规则 | JSON字段 | 用法 |
|------|----------|------|
| 短线看20日线 | `technical.above_ma20`, `technical.ma20_slope_up` | 两者都为true，短线趋势才初步成立 |
| 中线看60日线 | `technical.above_ma60`, `technical.ma60_slope_up` | 两者都为true，中线趋势才初步成立 |
| 大跌日禁追 | `engine_flags.big_down_day_over_5_pct` | true时不得给立即买入 |
| 追高降级 | `change_20d_pct`, `change_60d_pct` | 判断是否只能观察仓 |
| 止损幅度 | `technical.atr14_pct`, `technical.low_20d` | 结合结构止损与ATR取较大者 |
| 买点分类 | `recent_10d`, `ma20`, `ma60`, `high_20d`, `low_20d` | 判断回踩、突破、修复、左侧试错 |

仓位公式仍需用户账户风险参数。  
若用户未提供，决策引擎默认单笔可承受亏损为账户总额1%，高波动品种可降为0.5%。

## 10. 与自查手册的对应关系

| 自查手册步骤 | JSON/底稿支持 | 仍需人工填写 |
|--------------|---------------|--------------|
| 步骤一：界定投资周期 | `meta.requested_period` | 周期匹配检验 |
| 步骤二：基本面 | `financial.*` | 应收、存货、商誉、同行均值 |
| 步骤三：区分类型 | 行情和财务趋势 | 稳定/成长/周期/反转的最终判断 |
| 步骤四：回报率测算 | PE、PB、股息率、EPS | 目标PE、未来EPS、估值分位 |
| 步骤五：筹码与博弈 | 公告关键词 | 股东户数、融资、北向、解禁 |
| 步骤六：故事与逻辑 | 财务和公告辅助 | 三段式逻辑全文 |
| 步骤七：技术面 | `technical.*` | 平台结构、量能质量 |
| 步骤八：决策与仓位 | ATR、关键价位 | 账户风险、仓位上限、止损价 |
| 步骤九：持有中自检 | 当前价、财务更新 | 成本、仓位、原始买入逻辑 |
| 步骤十：卖出执行 | 技术位、证伪数据 | 止盈/止损执行计划 |

## 11. 后续脚本扩展优先级

为了让 JSON 更接近决策引擎的完整输入，建议按以下顺序扩展：

1. 近5年PE中位数和PE分位数。
2. 近10年PB分位数。
3. 应收账款、存货、商誉、净资产。
4. 股东户数变化。
5. 融资余额变化。
6. 北向资金变化。
7. 解禁日程。
8. 公告原文链接和公告日期过滤。
9. 自动生成通用底稿 Markdown。
10. 自动生成“数据缺口清单”和“需人工判断清单”。

---

## 12. 使用纪律

- 脚本输出的 true/false 只代表机械条件，不代表投资结论。
- 数据缺口必须写出来，不能用主观判断替代。
- 估值缺关键参数时，不能直接给“买入”。
- 公告关键词结果必须人工复核。
- 闸门F的核心逻辑、催化剂、证伪信号必须人工写清楚。
- 最终结论必须仍按决策引擎 A-G 顺序执行。
