# 天才交易员架构设计 v0.1

> 本文档用于指导后续分阶段开发。  
> 当前目标不是一次性实现完整系统，而是先把方向、边界、模块、数据流和开发顺序定清楚，后续每个阶段再拆成单独实现文档。

---

## 1. 项目定位

本项目是面向 A 股投资的本地天才交易员辅助系统。

核心目标：

- 自动抓取和整理股票数据。
- 按确定性规则判断买入、观察、持有、减仓、清仓。
- 维护真实持仓和模拟盘持仓。
- 记录每日决策、操作、持仓变化和账户表现。
- 通过 AI 生成解释、复核和报告。
- 后续通过插件输出提醒、报告或交易计划。

系统定位是“决策辅助系统”，不是第一阶段就自动下单的交易系统。

---

## 2. 核心原则

### 2.1 数据先行

所有结论必须来自结构化数据、持仓数据、交易规则和策略规则。

AI 不允许在缺少关键数据时直接给强买入、强卖出或强仓位建议。

### 2.2 规则优先

买入、卖出、仓位、止损、清仓先由规则引擎和风控引擎产出。

AI 的职责是：

- 解释规则结论。
- 复核明显矛盾。
- 总结风险。
- 生成报告。
- 辅助用户理解。

AI 不直接绕过规则做最终交易决策。

### 2.3 可追溯

每一条建议、每一次模拟交易、每一次真实操作都必须能追溯到：

- 当时的数据。
- 命中的规则。
- 未通过的规则。
- 策略版本。
- AI prompt 版本。
- 用户是否执行。
- 执行结果。

### 2.4 风控优先

宁可因为数据缺失而降级，也不要在不完整数据下给出激进结论。

所有买入建议必须经过风控模块拦截。

### 2.5 本地优先，逐步服务化

第一版使用：

```text
Python + SQLite + JSON 缓存 + Markdown 报告
```

后续按需要演进到：

```text
FastAPI + Web Dashboard + PostgreSQL + Redis + 任务队列
```

不在第一版引入不必要的复杂中间件。

### 2.6 策略透明可改

交易策略是本系统最核心的资产，不能做成 AI 黑盒。

要求：

- 策略规则必须以代码、配置和文档形式明确表达。
- 每条结论必须能解释命中的具体规则和数据。
- 策略参数必须能快速调整。
- 策略版本必须可追溯。
- AI 不能直接生成不可审计的买卖结论。
- 修改策略后必须能快速回放历史样本验证影响。

### 2.7 数据有效性优先

数据收集不是越多越好，只收集对规则、风控、复盘有明确用途的数据。

要求：

- 不采集无明确用途的数据。
- 不把低可信、滞后、口径不清的数据作为硬规则依据。
- 缺失数据必须标注，不允许凭空补。
- 异常数据必须识别并降级。
- 每个字段都应能映射到规则、报告、风控或复盘场景。

### 2.8 快速验证和快速修正

系统必须支持快速判断策略是否有效。

要求：

- 策略修改后能快速跑固定样本。
- 能对比修改前后的决策差异。
- 能在模拟盘中跟踪策略表现。
- 能复盘错误案例并沉淀为测试样本。
- 能区分“策略错了”“数据错了”“执行错了”“用户没执行”。

### 2.9 时间是第一等数据

交易系统中时间错误会导致所有结论失真。

要求：

- 所有行情、财报、公告、决策、订单、成交、持仓快照都必须带时间戳。
- 必须区分交易日、自然日、报告期、公告披露日、数据抓取时间和决策生成时间。
- 任何规则只能使用当时已经公开且已经生效的数据。
- 财报按公告披露时间生效，不按报告期提前生效。
- 公告按披露时间生效，盘后公告不能影响当日盘中决策。
- A 股 T+1 必须按交易日历处理，不能按自然日简单加一天。
- 盘前、盘中、盘后流程必须有明确时间边界。
- 所有回放和模拟盘必须使用历史当时视角，禁止未来函数。
- 所有时间统一保存标准格式，并记录本地交易时区。

## 3. 总体架构

系统按层拆分：

```text
数据源
  ↓
数据抓取与缓存层
  ↓
数据库与数据底稿层
  ↓
交易规则与资产规则层
  ↓
策略引擎
  ↓
风控引擎
  ↓
组合 / 持仓 / 模拟盘
  ↓
盘前 / 盘中 / 盘后工作流
  ↓
AI 解释与复核层
  ↓
报告 / CLI / Web / 插件输出
```

第一版必须保证核心逻辑可以脱离界面独立运行。

## 4. 模块划分

### 4.1 数据抓取模块 `data_fetching`

职责：

- 抓取行情。
- 抓取 K 线。
- 抓取财务。
- 抓取估值。
- 抓取公告。
- 抓取股东户数。
- 抓取融资融券。
- 抓取北向持仓。
- 抓取限售解禁。
- 生成单票结构化 JSON。
- 写入数据库缓存。

第一版沿用现有 `stock_decision_data.py` 能力，后续再拆模块。

关键要求：

- 数据抓取失败必须显式记录。
- 不得编造缺失数据。
- 每次数据写入必须有时间戳和来源。
- 财报按公告日生效，避免未来函数。

### 4.2 存储模块 `storage`

职责：

- 管理 SQLite 数据库。
- 管理 JSON 数据底稿。
- 管理 Markdown 报告。
- 管理缓存过期策略。

第一版建议：

- SQLite 作为主存储。
- JSON 作为原始数据底稿和调试材料。
- Markdown 作为人类阅读报告。

后续如出现多进程、多用户或大规模任务，再迁移 PostgreSQL。

### 4.2.1 数据质量模块 `data_quality`

职责：

- 判断数据是否完整。
- 判断数据是否可信。
- 判断数据是否新鲜。
- 判断数据是否能作为规则依据。
- 识别异常值。
- 输出数据质量评分。

数据分级：

```text
REQUIRED     必需数据，缺失则不能评估或必须降级
IMPORTANT    重要数据，缺失会降低信心或仓位
OPTIONAL     辅助数据，只用于解释或提示
NOISE        暂无明确用途的数据，不采集或不入库
```

数据可信度分级：

```text
PRIMARY      官方公告、交易所、财报原文等高可信数据
DERIVED      由基础数据计算出的指标，例如均线、涨幅、估值分位
THIRD_PARTY  第三方接口数据，需要标注来源和时间
STALE        日期滞后，只能做参考
UNKNOWN      来源或口径不清，不作为硬规则依据
```

处理规则：

- 缺失必需数据时，直接标注数据不足。
- 重要数据缺失时，结论降级。
- 可疑异常值必须标记，不得静默使用。
- 滞后数据只能作为参考，不能作为当日交易硬触发。
- 不同来源冲突时，优先使用高可信来源，并记录冲突。
- AI 不允许补造缺失字段。

### 4.2.2 有效数据目录 `data_catalog`

职责：

- 定义每个字段为什么采集。
- 定义字段来源。
- 定义刷新频率。
- 定义是否参与规则判断。
- 定义缺失时如何处理。

每个字段必须能回答：

- 用在哪条规则。
- 是否影响买入或卖出结论。
- 是否影响仓位。
- 是否影响风控。
- 是否只用于报告解释。
- 刷新频率是多少。
- 数据源失败时如何降级。

没有明确用途的数据，不进入第一版采集范围。

### 4.3 资产类型模块 `instruments`

职责：

- 统一管理股票、ETF、基金等标的。
- 区分不同资产类型的交易规则和策略规则。

第一版必须预留资产类型字段，但策略先只实现 A 股股票。

资产类型：

```text
A_STOCK   A 股股票，第一阶段重点
ETF       场内 ETF，第二阶段考虑
FUND      场外基金，第三阶段考虑
```

不同资产不能混用同一套策略。

### 4.4 A 股交易规则模块 `market_rules`

职责：

- 实现 A 股 T+1。
- 实现 100 股整数买入规则。
- 实现价格最小变动单位。
- 实现涨跌停规则。
- 实现停牌、ST、退市风险等交易限制。
- 管理交易日历。

模拟盘和未来实盘必须共用该模块。

重点规则：

- 当日买入股票当日不可卖出。
- 持仓需要区分总数量、可卖数量、锁定数量。
- 买入数量必须按 100 股整数倍。
- 跌停可能卖不出，涨停可能买不到。

### 4.5 策略引擎 `strategy_engine`

职责：

- 将现有投资决策文档代码化。
- 输入结构化数据，输出结构化结论。
- 规则结果必须可复现。
- 策略规则必须可读、可配置、可测试。
- 策略参数必须能在不改核心代码的情况下调整。

买入评估输出：

```text
买入
观察仓
观望
不买入
```

持仓评估输出：

```text
持有
减仓一半
清仓
```

策略引擎必须输出：

- 最终建议。
- 信心评级。
- 命中规则。
- 未通过规则。
- 数据缺口。
- 关键价位。
- 证伪点。
- 加仓条件。
- 减仓条件。
- 清仓条件。

策略版本必须记录。

策略引擎禁止事项：

- 禁止依赖 AI 直接输出最终买卖结论。
- 禁止只有自然语言解释、没有结构化规则命中。
- 禁止把缺失数据当作正常数据。
- 禁止无法复现的随机判断。
- 禁止修改策略后不记录版本。

### 4.5.1 策略验证模块 `strategy_lab`

职责：

- 快速验证策略是否有效。
- 对比策略修改前后的决策差异。
- 维护固定样本集。
- 回放历史决策。
- 分析模拟盘表现。
- 沉淀错误案例。

核心能力：

- 固定样本回归测试。
- 单票历史回放。
- 自选股池批量回放。
- 策略版本 A/B 对比。
- 决策差异报告。
- 错误案例归因。

归因类型：

```text
STRATEGY_ERROR     策略规则错误
DATA_ERROR         数据错误或缺失
EXECUTION_ERROR    执行偏差
USER_OVERRIDE      用户未执行或手动覆盖
MARKET_NOISE       市场随机波动
```

策略修改流程：

```text
提出规则修改
  ↓
更新策略配置或代码
  ↓
生成新策略版本
  ↓
跑固定样本回归
  ↓
跑模拟盘或历史回放
  ↓
输出差异报告
  ↓
确认后启用
```

### 4.6 风控模块 `risk_control`

职责：

- 对策略引擎输出做最终拦截。
- 控制仓位、风险事件和数据缺口。

建议规则：

- 单票仓位上限。
- 单行业仓位上限。
- 总权益仓位上限。
- 黑名单硬否决。
- 灰名单降级。
- 数据缺失降级。
- 大跌日禁止追买。
- 财报窗口期提示。
- 连续亏损后暂停买入。
- 模拟盘最大回撤限制。

风控模块可以把结论降级，但不能被 AI 绕过。

### 4.7 股票池模块 `watchlist`

职责：

- 管理自选股池。
- 管理买入观察池。
- 管理持仓池。
- 管理黑名单和灰名单。
- 支持后续行业池、主题池。

第一版先支持 CSV 或 SQLite 管理。

### 4.8 组合与持仓模块 `portfolio`

职责：

- 维护真实持仓。
- 维护成本价、仓位、持仓数量。
- 维护买入逻辑和原证伪点。
- 维护止损价和计划仓位。
- 记录每日持仓快照。

持仓必须包含：

- 总数量。
- 可卖数量。
- T+1 锁定数量。
- 成本价。
- 当前价。
- 当前市值。
- 浮盈亏。
- 仓位比例。
- 买入逻辑。
- 原证伪点。

没有持仓信息时，卖出评估只能标记为预评估。

补充约束：

- Web 页面允许维护买入逻辑、证伪点、止损价、目标价、计划仓位和备注。
- 这些维护属于“持仓计划信息”，不属于交易执行。
- 人工维护必须写入审计记录。
- 当前 v0.1 阶段，持仓计划字段允许作为 SQLite 本地覆盖层，避免被空 JSON 导入覆盖。
- 不允许在持仓维护页面修改现金、成本、数量、可卖数量和锁定数量。

### 4.9 模拟盘模块 `paper_trading`

职责：

- 用虚拟资金执行天才交易员建议。
- 验证策略和执行质量。
- 记录交易和账户表现。

必须支持：

- 初始资金。
- 现金。
- 持仓市值。
- 总资产。
- 买入、卖出、减仓、清仓。
- 佣金、印花税、滑点。
- A 股 T+1。
- 涨跌停成交限制。
- 每日资产快照。
- 最大回撤。
- 胜率。
- 盈亏比。
- 与基准指数对比。

模拟盘不能使用未来数据。

### 4.10 工作流模块 `workflow`

职责：

- 组织盘前、盘中、盘后的交易日流程。

#### 盘前流程

目标：提前生成今日交易计划。

动作：

- 处理 T+1 持仓解锁。
- 更新昨晚公告。
- 更新昨日行情和技术指标。
- 扫描持仓。
- 扫描自选股。
- 生成今日买入观察。
- 生成今日卖出风险。
- 生成关键触发价。

#### 盘中流程

目标：只处理触发，不做复杂慢判断。

动作：

- 监控价格触发。
- 监控止损触发。
- 监控突破触发。
- 监控跌破均线。
- 监控涨跌停。
- 生成提醒。

第一版盘中只做简单触发提醒。

#### 盘后流程

目标：归档、复盘、更新次日准备数据。

动作：

- 更新收盘数据。
- 计算技术指标。
- 更新模拟盘成交。
- 写入每日持仓快照。
- 写入账户快照。
- 评估当日建议是否执行。
- 生成盘后复盘报告。

### 4.11 AI 模块 `ai`

职责：

- 调用 AI 接口。
- 管理 prompt 模板。
- 管理输出 JSON Schema。
- 生成自然语言报告。
- 复核规则结论。

AI 可以做：

- 报告生成。
- 风险总结。
- 公告摘要。
- 财报摘要。
- 决策解释。
- 用户问答。

AI 不可以做：

- 绕过硬规则。
- 覆盖风控结论。
- 在数据缺失时强行给强结论。
- 未经用户确认自动下单。
- 自行修改策略参数并立即生效。

AI 调用必须记录：

- 调用时间。
- 模型名称。
- prompt 版本。
- 输入摘要。
- 输出结果。
- token 和成本。
- 是否参与最终报告。
- 关联 decision_id。

### 4.12 报告模块 `reporting`

职责：

- 生成机器可读 JSON。
- 生成用户可读 Markdown。
- 后续支持插件输出。

报告类型：

- 单票买入评估。
- 单票卖出评估。
- 盘前计划。
- 盘中触发提醒。
- 盘后复盘。
- 模拟盘绩效报告。
- 策略回放报告。

### 4.13 接口层 `interfaces`

职责：

- CLI 命令入口。
- 后续 Web API。
- 后续插件输出。

第一版建议先做 CLI：

```text
ai-trader evaluate 002563 --side sell
ai-trader scan --watchlist default
ai-trader pre-market
ai-trader intraday
ai-trader post-market
ai-trader paper-run
```

第二阶段再做本地 Web Dashboard。

### 4.14 界面模块 `ui`

第一版不做复杂界面。

后续可做本地 Web：

- 总览 Dashboard。
- 今日盘前计划。
- 持仓列表。
- 买入观察池。
- 卖出提醒。
- 模拟盘账户。
- 决策记录。
- 单票详情。
- 策略配置。
- 数据抓取状态。

### 4.15 配置模块 `config`

职责：

- 管理策略参数。
- 管理风控参数。
- 管理数据源配置。
- 管理账户配置。
- 管理 AI 模型配置。
- 管理运行环境配置。

建议配置文件：

```text
config/strategy_rules.yaml
config/risk_rules.yaml
config/data_sources.yaml
config/accounts.yaml
config/runtime.yaml
```

敏感信息只允许放在 `.env` 或系统环境变量中，不允许进入 Git。

敏感信息包括：

- AI API Key。
- 数据源 Token。
- 未来券商插件凭证。
- 通知插件凭证。

### 4.16 任务与日志模块 `runtime`

职责：

- 管理任务执行记录。
- 管理失败重试。
- 管理运行日志。
- 管理数据抓取状态。
- 管理定时任务状态。

所有自动任务都必须记录：

- 任务名称。
- 开始时间。
- 结束时间。
- 输入参数。
- 成功或失败。
- 失败原因。
- 影响的股票。
- 生成的报告或决策 ID。

第一版可以只使用本地日志和 SQLite 表记录。

后续如引入任务队列，再把同一套任务模型迁移到队列执行。

### 4.17 资金与账户模块 `accounts`

职责：

- 管理模拟盘账户。
- 记录实盘账户资金快照。
- 管理本金规模。
- 管理可用现金。
- 管理冻结资金。
- 管理资金流水。
- 管理单账户、单策略、单资产类型的资金上限。

核心原则：

- 真实资金不存放在本系统中，只存在券商账户或银行账户中。
- 本系统只保存实盘账户的资金快照、资金流水记录和计划投入额度。
- 模拟盘资金是虚拟资金，可以完整保存在 SQLite 中。
- 所有买入建议必须经过可用资金校验。
- 本金有限时，系统必须优先输出“买多少”和“不买什么”，而不是只输出“可以买”。

资金字段至少区分：

- 初始本金。
- 可用现金。
- 冻结资金。
- 持仓市值。
- 总资产。
- 今日可买额度。
- 单票最大投入金额。
- 单日最大买入金额。
- 现金保留比例。
- 已占用风险预算。

资金约束必须进入风控模块：

- 可用资金不足时，买入建议自动降级为观察。
- 超过单票资金上限时，自动缩小建议仓位。
- 现金低于保留比例时，暂停新增买入。
- 多只股票同时触发买入时，按优先级分配资金。
- 模拟盘和实盘账户必须分开计算，不能混用本金。

### 4.18 时间与交易日历模块 `timekeeper`

职责：

- 统一处理系统时间。
- 统一处理 A 股交易日历。
- 判断盘前、盘中、盘后。
- 判断数据是否在决策时已经生效。
- 判断 T+1 可卖日期。
- 为回放和模拟盘提供历史时点视角。

必须区分的时间：

- `trade_date`：交易日。
- `calendar_date`：自然日。
- `report_period`：财报报告期。
- `announce_time`：公告披露时间。
- `data_observed_at`：系统抓取到数据的时间。
- `decision_time`：生成决策的时间。
- `order_time`：委托时间。
- `trade_time`：成交时间。
- `snapshot_time`：账户或持仓快照时间。
- `effective_from`：数据开始允许被策略使用的时间。

时间处理规则：

- 所有时间字段必须带明确语义，不能只用模糊的 `date`。
- A 股交易逻辑默认使用 `Asia/Shanghai`。
- 保存时建议使用 ISO 8601 字符串，并记录时区。
- 回放时必须把系统时钟固定在历史决策时间。
- 数据没有 `effective_from` 时，不允许作为硬规则依据。
- 盘前计划不能使用当天收盘后才产生的数据。
- 盘后复盘可以使用当天收盘数据，但不能反向影响当天盘中决策。

## 6. 数据流

### 6.1 单票买入评估

```text
输入股票代码
  ↓
抓取或读取缓存数据
  ↓
写入 raw_data 原始数据层
  ↓
生成 normalized_data 标准化数据
  ↓
通过 data_quality 和 timekeeper 检查
  ↓
生成 strategy_snapshot 策略输入快照
  ↓
策略引擎买入评估
  ↓
风控引擎拦截
  ↓
账户资金校验和仓位换算
  ↓
记录 decision_result
  ↓
AI 生成解释
  ↓
输出 Markdown / JSON
```

### 6.2 持仓卖出评估

```text
读取持仓
  ↓
读取当前行情和技术指标
  ↓
读取原买入逻辑和证伪点
  ↓
生成 strategy_snapshot 策略输入快照
  ↓
策略引擎持仓评估
  ↓
风控引擎复核
  ↓
记录 decision_result
  ↓
模拟盘或用户执行
  ↓
记录 trade_record
```

### 6.3 每日盘前

```text
交易日校验
  ↓
T+1 持仓解锁
  ↓
更新公告和数据
  ↓
读取账户资金和可用买入额度
  ↓
扫描持仓和自选股
  ↓
生成今日交易计划
  ↓
记录 daily_plan
  ↓
输出盘前报告
```

### 6.4 每日盘后

```text
更新收盘行情
  ↓
更新技术指标
  ↓
执行模拟盘结算
  ↓
写入持仓快照
  ↓
写入账户快照
  ↓
评估当日决策表现
  ↓
生成盘后复盘
```

### 6.5 任务执行形态

第一版采用本地顺序执行：

```text
CLI 命令
  ↓
读取配置
  ↓
执行任务
  ↓
写入 SQLite
  ↓
输出 JSON / Markdown
```

第二阶段加入本地定时任务：

```text
APScheduler
  ↓
盘前任务 / 盘后任务 / 数据刷新任务
  ↓
workflow_runs 记录状态
```

第三阶段如需要并发或插件消费，再拆成任务队列：

```text
任务生产者
  ↓
Redis / 队列
  ↓
Worker 执行数据抓取、策略评估、报告生成
  ↓
数据库保存结果
  ↓
插件消费结构化结果
```

任务设计要求：

- 每个任务尽量幂等。
- 任务输入和输出都要可序列化。
- 任务结果以数据库为准，不依赖内存状态。
- 插件只消费结果，不重新计算策略。
- 分布式执行只能提升吞吐，不能改变交易结论。

## 10. 质量保障与验收要求

### 10.1 测试分层

后续开发至少包含：

- 单元测试：规则函数、风控函数、交易规则。
- 集成测试：数据抓取到决策输出的完整链路。
- 回归测试：典型股票样本的固定输出。
- 模拟盘测试：买入、卖出、减仓、清仓、T+1。
- 数据质量测试：缺字段、接口失败、异常值。
- 未来函数测试：财报和公告只能在披露日之后生效。
- 时间一致性测试：交易日、公告时间、决策时间和成交时间不能冲突。
- 策略差异测试：策略修改前后输出差异必须可见。
- 数据有效性测试：无用途字段不进入第一版采集范围。

### 10.2 样本用例

建议维护一批固定测试样本：

- 趋势向上且基本面正常的股票。
- 大涨后追高风险股票。
- 跌破 20 日线的短线持仓。
- 跌破 60 日线的中线持仓。
- 扣非连续亏损股票。
- 现金流异常股票。
- 有减持或监管公告股票。
- 当日买入后触发止损但因 T+1 不可卖出的股票。

### 10.3 验收口径

每个阶段验收时必须回答：

- 输入是什么。
- 输出是什么。
- 数据存在哪里。
- 决策是否可追溯。
- 失败时如何降级。
- 是否影响现有流程。
- 是否违反 A 股交易规则。
- 是否引入未来函数。
- 时间字段是否完整且语义明确。
- 数据生效时间是否早于决策时间。
- 是否依赖 AI 黑盒判断。
- 是否存在凭空补数据。
- 是否采集了无明确用途的数据。
- 策略修改后是否能快速回放验证。

### 10.4 可观测性

系统运行必须能看见：

- 哪些任务执行成功。
- 哪些任务失败。
- 哪些数据缺失。
- 哪些结论被降级。
- 哪些建议被用户执行。
- 哪些建议未执行。
- 模拟盘和真实执行之间有什么差异。

## 11. 风险与约束

### 11.1 数据风险

- 免费接口可能失败。
- 公告接口可能滞后。
- 北向数据可能不是实时。
- 历史估值可能需要人工复核。

处理方式：

- 数据缺失显式标注。
- 关键数据缺失时降级。
- 报告中列出数据缺口。

### 11.2 策略风险

- 规则可能过拟合。
- 估值逻辑可能不适合所有行业。
- 趋势规则可能频繁误触发。

处理方式：

- 所有规则版本化。
- 模拟盘长期验证。
- 定期复盘误判案例。

### 11.3 AI 风险

- AI 可能幻觉。
- AI 可能解释得合理但结论错误。
- AI 可能忽略数据缺口。

处理方式：

- AI 只做解释和复核。
- AI 输出必须结构化校验。
- AI 不得绕过硬规则和风控。

### 11.4 执行风险

- 模拟盘成交与真实市场不同。
- 涨跌停、滑点、流动性会影响实际执行。
- 用户执行可能与建议不同。

处理方式：

- 记录建议和执行差异。
- 模拟盘加入滑点和成交限制。
- 实盘前必须人工确认。

## 14. 目标、差距与推进边界

### 14.1 长期目标

“天才交易员”的长期目标不是简单的 AI 问答工具，而是逐步演进为小型量化交易系统。

长期目标包括：

- 数据可信。
- 时间正确。
- 策略透明。
- 决策可复现。
- 回测严谨。
- 模拟盘可验证。
- 持仓可管理。
- 风险可拦截。
- 执行可追溯。
- 错误可复盘。
- 策略可迭代。

目标系统最终应该能回答：

- 为什么买。
- 为什么不买。
- 为什么卖。
- 买多少。
- 卖多少。
- 什么条件下加仓。
- 什么条件下止损。
- 这条规则过去表现如何。
- 这次失败是策略问题、数据问题、执行问题还是市场噪声。

### 14.2 当前定位

当前阶段定位为：

```text
个人量化决策辅助系统
```

第一版做出来的价值是：

- 自动整理数据。
- 规则化判断买卖。
- 管理持仓和本金。
- 记录决策和执行。
- 用模拟盘验证策略。
- 生成盘前计划和盘后复盘。
- 帮助用户减少随意交易。

第一版不追求：

- 自动赚钱。
- 全市场高频量化。
- 机构级多因子平台。
- 全自动实盘下单。
- AI 黑盒自主交易。

### 14.3 与专业量化系统的差距

当前架构距离完整专业量化系统还有以下差距。

#### 研究体系不足

专业量化需要完整 research loop：

```text
假设
  ↓
数据
  ↓
因子 / 信号
  ↓
回测
  ↓
归因
  ↓
风险评估
  ↓
上线
  ↓
监控
  ↓
下线或迭代
```

后续需要补：

- 因子定义。
- 信号生成。
- 样本内 / 样本外验证。
- 参数敏感性测试。
- 过拟合检查。
- Benchmark 对比。
- 收益归因。

#### 回测系统不足

模拟盘不能替代严谨回测。

后续需要补：

- 点时数据。
- 前复权 / 后复权口径。
- 停牌处理。
- 涨跌停不可成交处理。
- 滑点模型。
- 佣金和印花税。
- 最小交易单位。
- T+1。
- 生存者偏差处理。
- 财报公告日生效。
- 调仓频率。
- 资金曲线。
- 最大回撤。
- 夏普、卡玛、波动率、胜率、盈亏比。

#### 数据体系不足

后续需要把数据分层：

```text
原始数据层
  ↓
清洗数据层
  ↓
特征 / 指标层
  ↓
决策数据层
  ↓
回测快照层
```

每层都要有：

- 来源。
- 时间。
- 口径。
- 质量检查。
- 缺失处理。
- 生效时间。

#### 组合管理不足

当前更多是单票判断，后续必须补组合构建。

需要回答：

- 多只股票同时可以买时买谁。
- 每只买多少。
- 行业是否过度集中。
- 当前现金够不够。
- 新信号和已有持仓冲突怎么办。
- 是否等权、按信号强度、按风险预算分配。
- 组合回撤时是否整体降仓。

#### 执行系统不足

当前只适合模拟盘和人工确认。

未来实盘前需要补：

- 订单状态机。
- 委托、撤单、部分成交。
- 成交回报。
- 账户对账。
- 异常重试。
- 券商接口失败处理。
- 防重复下单。
- 人工确认。
- 急停开关。

#### 风控分层不足

后续风控要拆成：

- 交易前风控：能不能买、买多少。
- 交易中风控：是否触发止损、是否超仓。
- 交易后风控：回撤、集中度、策略失效。
- 系统风控：数据失败、时间错误、接口异常。

### 14.4 资源有限下的推进原则

因为本金、时间、数据源和开发资源都有限，推进必须遵守：

- 先小范围闭环，不做大而全。
- 先自选股和持仓，不做全市场扫描。
- 先收盘级策略，不做高频盘中策略。
- 先模拟盘，不接自动实盘。
- 先规则透明，不做 AI 黑盒。
- 先有效数据，不抓无用数据。
- 先验证策略，再扩大资金和范围。
- 先记录错误，再修正规则。

### 14.5 优先补齐顺序

后续按下面顺序逐步补齐：

1. 有效数据目录：明确哪些数据必须采集、为什么采集、怎么校验。
2. 策略与信号：把买入、卖出、仓位、止损规则彻底结构化。
3. 时间系统：确保交易日、公告日、决策时间、数据生效时间正确。
4. 回测与模拟盘：严谨验证策略，不允许未来函数。
5. 持仓和资金：把成本、仓位、本金、可用现金纳入决策。
6. 组合构建：解决多只股票同时触发时买谁、买多少。
7. 风控分层：从单票风控扩展到组合和系统风控。
8. 报告和复盘：输出可读报告和可机器分析的 JSON。
9. AI 解释和复核：AI 只做解释、摘要和异常复核。
10. 插件和界面：在核心闭环稳定后再做展示和输出。
11. 实盘前置：最后才考虑交易插件和人工确认下单。

### 14.6 判断是否偏离方向

如果后续开发出现以下情况，说明方向偏了：

- AI 可以绕过规则直接给买卖结论。
- 交易结论无法追溯到具体数据和规则。
- 数据缺失但系统仍给强结论。
- 策略修改后无法回放验证。
- 只做界面和报告，不做策略验证。
- 只看单票，不考虑本金和组合。
- 盘后数据影响了当天盘中决策。
- 模拟盘不遵守 T+1、涨跌停和滑点。
- 抓了大量数据但不知道服务哪条规则。
- 没有记录建议和真实执行之间的差异。

### 14.7 当前最重要的三件事

短期最重要的是：

1. 把数据边界定清楚。
2. 把策略规则结构化。
3. 把验证闭环做起来。

这三件事完成之前，不应优先投入复杂 UI、插件、自动交易或全市场扫描。

## 15. 赚钱目标与个人能力边界

### 15.1 最终目标

本系统最终目的就是赚钱。

但这里的“赚钱”不是指某一笔交易猜对，也不是短期收益截图，而是：

- 扣除交易成本后的净收益为正。
- 长期收益能跑赢合适基准。
- 最大回撤在可承受范围内。
- 策略失效时能及时识别并停止。
- 本金有限时能优先保证生存。
- 收益来源能被解释、验证和复盘。

系统不追求“每次都对”，而追求：

```text
长期正期望
风险可控
错误可修正
本金可持续
```

### 15.2 赚钱不能只看胜率

衡量策略是否真的有用，不能只看命中率。

必须同时看：

- 总收益率。
- 年化收益率。
- 最大回撤。
- 回撤修复时间。
- 胜率。
- 盈亏比。
- 单票最大亏损。
- 连续亏损次数。
- 资金利用率。
- 换手率。
- 交易成本。
- 相对基准收益。

高胜率但亏大钱的策略无效。

低胜率但盈亏比足够好的策略可能有效。

### 15.3 个人投资能力边界

系统可以帮助用户减少错误，但不能替代投资者的基本认知。

用户至少需要理解：

- 自己买的是什么。
- 钱为什么会赚。
- 什么情况说明判断错了。
- 最坏能亏多少。
- 亏损后是否还能继续执行规则。
- 当前策略适合短线、中线还是长线。
- 当前收益是能力、行情还是运气。

如果用户无法解释买入逻辑和退出条件，系统应降低结论强度。

### 15.4 个人炒股学习清单

用户后续需要逐步补齐以下知识。

#### 交易规则

- A 股 T+1。
- 涨跌停。
- 集合竞价。
- 交易时间。
- 停牌和复牌。
- ST 和退市风险。
- 主板、创业板、科创板规则差异。
- 买入单位和最小价格变动。

#### 财报和会计

- 营收。
- 归母净利润。
- 扣非净利润。
- 经营现金流。
- 应收账款。
- 存货。
- 商誉。
- 资产负债率。
- 毛利率、净利率、ROE。
- 一次性收益和非经常性损益。

#### 估值

- PE。
- PB。
- 股息率。
- 估值分位。
- 周期股估值。
- 成长股估值。
- 高股息资产估值。

#### 行业和周期

- 行业景气度。
- 政策影响。
- 产品价格周期。
- 库存周期。
- 产能周期。
- 行业竞争格局。

#### 风控和仓位

- 单票仓位上限。
- 总仓位控制。
- 止损。
- 止盈。
- 加仓条件。
- 清仓条件。
- 连续亏损后的暂停规则。

#### 概率和统计

- 胜率。
- 盈亏比。
- 期望收益。
- 最大回撤。
- 样本数量。
- 过拟合。
- 幸存者偏差。
- 未来函数。

#### 行为金融

- 追涨杀跌。
- 亏损死扛。
- 盈利过早卖出。
- 频繁交易。
- 仓位失控。
- 听消息交易。
- 不愿承认错误。

#### 信息识别

- 公告原文。
- 财报数据。
- 研报假设。
- 新闻标题。
- 市场传闻。
- 数据源口径。
- 信息披露时间。

### 15.5 系统对赚钱目标的约束

为了服务赚钱目标，系统必须坚持：

- 不为了给建议而给建议。
- 不为了提高交易频率而交易。
- 不为了追热点破坏风控。
- 不为了短期收益忽略最大回撤。
- 不因为 AI 解释合理就放弃规则验证。
- 不因为模拟盘赚钱就直接上实盘。
- 不因为一次交易成功就认为策略有效。
- 不因为一次交易失败就否定全部策略。

赚钱目标必须通过数据验证，而不是通过主观感觉确认。

### 15.6 第一阶段赚钱目标的现实口径

第一阶段不以实盘收益作为唯一验收标准。

第一阶段更现实的目标是：

- 少犯明显错误。
- 减少无计划交易。
- 发现高风险持仓。
- 建立买卖纪律。
- 用模拟盘验证策略。
- 记录每次决策和结果。
- 找到真正有效和无效的规则。

当模拟盘和回放连续证明策略有正期望后，才考虑逐步提高真实资金参与度。

## 16. AI 如何弥补用户知识缺口

### 16.1 基本定位

用户可以只提供“想干什么”和个人约束，例如：

- 想买什么类型的股票。
- 想做短线还是中线。
- 能承受多大亏损。
- 本金规模是多少。
- 不想碰哪些行业。
- 当前看好或担心什么。

AI 的价值是把这些模糊想法转成：

- 可验证假设。
- 数据需求。
- 候选规则。
- 风险清单。
- 反方观点。
- 学习材料。
- 模拟验证方案。

但 AI 不能把未经验证的想法直接变成实盘交易动作。

### 16.2 AI 的正确职责

AI 可以承担以下角色。

#### 投资助教

用于补齐用户知识短板。

输出：

- 概念解释。
- 财报字段解释。
- 估值方法解释。
- 行业逻辑解释。
- 交易规则解释。
- 常见错误提醒。

#### 策略研究助手

用于把用户想法拆成策略假设。

示例：

```text
用户想法：我想买低估值高股息股票。

AI 拆解：
- 什么是低估值。
- 股息率最低多少。
- 分红是否可持续。
- 是否排除周期顶部利润。
- 是否需要现金流覆盖分红。
- 买入后什么情况证伪。
```

#### 数据需求分析员

用于判断一个策略需要哪些数据。

输出：

- 必需字段。
- 重要字段。
- 可选字段。
- 无效字段。
- 数据来源。
- 更新频率。
- 缺失时如何处理。

#### 风险审查员

用于找漏洞，而不是鼓励交易。

必须检查：

- 是否数据不足。
- 是否逻辑跳跃。
- 是否高估确定性。
- 是否忽略仓位。
- 是否忽略时间。
- 是否有未来函数。
- 是否有行业或政策风险。
- 是否和已有持仓冲突。

#### 复盘教练

用于交易后复盘。

输出：

- 这次亏损是策略问题、数据问题、执行问题还是市场噪声。
- 当时哪些信号被忽略。
- 哪些规则需要修改。
- 哪些错误要沉淀为测试样本。

### 16.3 AI 输出必须分级

AI 输出不能只有“可以买”或“不能买”。

必须分成：

```text
知识解释
策略假设
候选规则
待验证结论
已验证规则
实盘建议
```

其中：

- `知识解释` 不产生交易动作。
- `策略假设` 只进入研究池。
- `候选规则` 只能进入回测或模拟盘。
- `待验证结论` 不能直接用于实盘。
- `已验证规则` 才能进入策略引擎。
- `实盘建议` 必须经过规则、数据、风控、资金和人工确认。

### 16.4 从用户想法到策略规则的流程

标准流程：

```text
用户提出想法
  ↓
AI 追问关键约束
  ↓
AI 拆成投资假设
  ↓
AI 列出需要的数据
  ↓
AI 给出候选规则
  ↓
系统检查数据是否可得
  ↓
回测或固定样本验证
  ↓
模拟盘运行
  ↓
复盘结果
  ↓
确认后进入正式策略
```

任何跳过验证直接实盘的流程，都视为方向偏离。

### 16.5 AI 必须主动暴露不确定性

AI 每次辅助用户决策时，必须输出：

- 当前结论依赖哪些假设。
- 哪些数据缺失。
- 哪些数据可能滞后。
- 哪些情况会导致判断失效。
- 最坏情况是什么。
- 是否需要人工确认。
- 是否只适合模拟盘验证。

如果 AI 无法说明不确定性，该输出不能用于交易。

### 16.6 AI 弥补知识缺口的边界

AI 可以帮助用户少走弯路，但不能消除市场不确定性。

AI 不能保证：

- 每次判断正确。
- 预测短期涨跌。
- 替用户承担亏损。
- 自动识别所有虚假信息。
- 在数据缺失时得出可靠结论。

AI 能帮助：

- 把问题问完整。
- 把逻辑拆清楚。
- 把数据列出来。
- 把风险摆出来。
- 把规则写清楚。
- 把结果复盘出来。

### 16.7 系统内置的 AI 工作模式

后续可在系统中内置几种 AI 模式。

```text
TUTOR          投资助教模式
RESEARCHER     策略研究模式
RISK_REVIEWER  风险审查模式
DATA_AUDITOR   数据审计模式
REPORTER       报告生成模式
REPLAY_COACH   复盘教练模式
```

默认模式不应是“交易员直接下结论”，而应是：

```text
先教育
再拆解
再验证
再决策
```

### 16.8 对用户知识不足时的保护规则

当用户无法提供完整买入逻辑、卖出条件、仓位计划或证伪点时：

- 系统可以帮助生成候选逻辑。
- 系统必须标注为“待验证”。
- 不允许直接给强买入。
- 不允许给大仓位。
- 默认只能进入观察池或模拟盘。

只有当逻辑、数据、风控和验证结果都明确后，才允许提升为正式策略。

### 17.3 v0.1 阶段定位

当前所有 `v0.1` 功能的核心价值是：

```text
先把完整交易辅助流程跑通
```

也就是说，当前阶段重点不是证明策略已经能赚钱，而是确认下面这些链路可以稳定串起来：

```text
数据底稿
  -> 策略快照
  -> 单票决策
  -> 组合计划
  -> 风控检查
  -> 模拟盘
  -> 盘前/盘中/盘后工作流
  -> 历史回放
  -> 报告复盘
```

后续真正有用的核心不是继续堆功能，而是进入策略迭代闭环：

```text
提出可解释策略规则
  -> 用历史回放快速验证
  -> 用模拟盘持续观察
  -> 对比胜率、盈亏比、回撤、持仓周期和资金占用
  -> 找出错误交易和漏掉的机会
  -> 修改策略规则、风控规则和仓位规则
  -> 再回放、再模拟
```

只有这个闭环能逐步回答：

- 哪些买入条件真的有效。
- 哪些卖出规则能保护本金。
- 哪些信号只是噪音。
- 哪些数据对决策有用，哪些数据会浪费资源。
- 当前策略适合什么市场环境。
- 本金有限时，应该优先买什么、放弃什么。

所以后续开发必须围绕一个核心标准推进：

```text
每个新增功能都要帮助更快验证、修正、复用交易策略。
```

不能为了“看起来更智能”而把系统变成黑盒，也不能为了增加功能而偏离赚钱目标。  
历史回放和模拟盘不是辅助模块，而是策略能否变强的验证机器。

### 17.4 当前最重要的缺口

下一步最重要的缺口不是 AI，也不是实盘自动交易，而是：

```text
策略有效性验证
SQLite 主账本
第一版可用界面
完整风控分层
```

原因：

- v0.1 已经证明流程可以跑通，并已补入最小策略调参记录。
- 现在已经能处理多个 `decision_result`，但策略强度还很初级。
- 多只股票同时触发时，已经能生成第一版组合计划，但行业集中度和相关性还没有做完整。
- 风控已经有第一版硬闸门，但还没有覆盖完整交易限制和异常行情。
- 模拟盘已经能被手动工作流驱动，盘中触发已有手动单次扫描，但还没有自动定时调度。
- 历史回放已有 `REPLAY_LITE`，但还需要更真实的数据源、点时财报和更完整成交假设。
- JSON 账本可以验证流程，但长期复盘和查询需要 SQLite。
- 现在命令行已经能用，后续需要一个第一版界面降低使用门槛。
